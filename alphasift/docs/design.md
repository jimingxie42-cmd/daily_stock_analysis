# 设计原则

## 1. 硬条件必须由代码执行

估值、流动性、涨跌幅、技术指标阈值等确定性条件全部由代码执行。不把可验证逻辑交给 LLM。

## 2. 默认规则必须可覆盖

系统可以提供开箱即用的默认评分曲线和风险阈值，但不能把市场风格假设写死在代码里。策略 YAML 可以通过 `scoring_profile`、`risk_profile`、`portfolio_profile` 和 `scorecard_profile` 覆盖默认行为。

## 3. 策略知识只维护一份

所有选股逻辑通过 `strategies/*.yaml` 的 `screening:` 段定义。不在代码中写死筛选条件。

## 4. 横向评分独立于单股诊断

`screen_score` 是选股专用的横向评分，不等于单股分析中的 `signal_score` 或 `sentiment_score`。

原因：
- 单股诊断分数为"这只票值不值得分析"设计，不是"全市场候选怎么比较"
- 直接复用会造成 L1/L2/L3 职责混叠

## 5. 排序层、风险层分开

alpha 排序和风险约束不揉成一个总分黑盒。风险覆盖层单独判定 veto/penalty。

## 6. LLM 只做软判断

LLM 负责：新闻语义理解、候选间相对排序、候选行业/主题标签、潜在催化、风险摘要、排序理由生成。
LLM 不负责：全市场初筛、数值阈值判断、精确技术指标计算。

## 7. 组合约束来自 LLM 但不交给 LLM 自由发挥

LLM 可以识别候选之间的行业/主题共性，并输出组合集中风险；如果候选自带 `industry/concepts/board_heat_score/board_heat_trend_score/board_heat_persistence_score/board_heat_cooling_score`，这些结构化字段会作为 LLM 判断锚点。代码层把 LLM 标签或结构化行业映射到行业风险桶，再转成可审计的 `portfolio_penalty`。例如银行、券商、保险同属金融风险桶。这样既利用 LLM 的语义归类能力，又避免让 LLM 直接决定硬性行业配额。

行业/概念/板块热度锚点优先来自本地 CSV/JSON/JSONL 映射文件，也可显式启用 AkShare 板块反查。默认不自动抓全市场板块，以免一次选股变成大量网络请求。

## 8. 先 A 股 P0，再谈多市场

P0 只面向 A 股。港股/美股等到字段映射和数据源确认后再扩展。

## 9. 优雅降级

- 全市场快照不可用 → 任务失败（fail-fast）
- 关键字段缺失 → 任务失败（不做模糊降级）
- 策略必需的日 K 增强单行失败 → 记录 degradation 并让该候选在日 K 硬筛中淘汰；全局日 K 增强不可运行才任务失败
- 可选日 K 增强失败 → 回退到快照评分并记录 degradation
- LLM 排序失败 → 回退到 screen_score 排序
- 可选 L3 后置分析器失败 → 当前任务失败或记录 degradation，取决于分析器是否为用户显式请求

## 10. 与外部分析系统解耦

alphasift 负责 L1（硬筛）和 L2（排序）。L3 是后置分析器框架，默认调用本地 scorecard；也可追加外部 DSA，或其他 HTTP 评分工具。

这里的 DSA 指外部项目 `daily_stock_analysis`：
- alphasift 负责全市场候选发现和排序
- daily_stock_analysis 负责单股深度分析
- 两者通过 `DSA_API_URL` 解耦部署

为控制成本，默认 scorecard 覆盖全部最终候选；DSA 只在最终入围候选的前 N 只上调用。DSA 不是全市场筛选的核心依赖，也不是唯一 L3 工具。

## 11. 选股必须可评估

一次选股运行必须能保存足够上下文，便于后续评估：

- 策略名与 `strategy_version`
- 数据源与降级记录
- L1/L2/L3 分数和风险字段
- 候选保存时价格

当前 T+N 评估用保存价与评估时最新快照价计算收益、胜率、缺失报价、交易成本扣减和突破/回踩形态后验标签。可选价格路径会额外估算最大回撤和最大浮盈。它是工程闭环，不是完整复权回测。
